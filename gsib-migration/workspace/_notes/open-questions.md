## A. Scope & Schema Disambiguation

### A1 — `?` U — What is "PRI1" in the v0.01 Inventory's Source Schema column?
The v0.01 inventory uses three values for Source Schema: `MNETD`, `PRI1`, `MNETD & PRI1`. Phase 1 grounding D3 confirms **MNETD = MNET** (legacy backend Oracle DB). But **PRI1 is undefined** in any decision/source we have. Hypothesis: PRI1 = the eAuction/eApps front-end primary database (separate Oracle from MNETD), reached via the `MNET` DB-link in `script-04-FE_TRI1.json`. **Need:** confirm whether `PRI1` is an alias for **TRI1** (the dm-tool script name) or a **third** database we haven't received DDL for.
- Asked: 2026-05-05
- Affects: every Inventory row's `Source Schema` value; the `migration scope` table count

### A2 — `?` U — Is `script-04-FE_TRI1.json` the same system the inventory calls `PRI1`?
Both source scripts (BE_MNETD, FE_TRI1) parse to the Oracle schema name `MS9ABA`, but inventory tags rows with logical schema names `MNETD`, `PRI1`, `MNETD & PRI1`. We need a mapping {script file ↔ inventory schema label}. Hypothesis: BE_MNETD = `MNETD`; FE_TRI1 = `PRI1`; rows tagged `MNETD & PRI1` = same logical table replicated into both DBs (e.g. `ABA0001_SECURITY_MASTER`).
- Asked: 2026-05-05
- Affects: every mapping row's source-side correctness

### A3 — `?` NLM — Does an FSD describe the `MNETD ↔ PRI1` data flow?
Looking for: which DB owns which write, are they kept in sync via DB Link / triggers / batch jobs, what's the canonical source-of-truth for tables that exist on both. Specifically `ABA0001_SECURITY_MASTER` since it's the central legacy entity the spec migrates to multiple OMEGA targets.
- Asked: 2026-05-05
- Affects: which copy to read from in the actual DMS extract

## B. Migration Modules & Sequencing

### B1 — `?` U — How many modules should v0.02 cover?
v0.01 ships 11 modules (SECMST x2, ISSCAL x4, ISSANN x1, BIDCOL x3). After D7 (`sec` folded into `iss`), `MAS-OMEGA-SECMST-001` becomes `MAS-OMEGA-ISSMST-001` (or similar) targeting `iss.iss_security_master`. **Need:** a target list of v0.02 modules. Proposal:
- Keep all 11 v0.01 modules (rename SECMST sheets if needed).
- Add modules covering the rest of the R1 surface: cm.cm_bank_master, cm.cm_applicant, cm.cm_counterparty, cm.cm_master_code, cm.cm_master_code_category, cm.cm_public_holiday, cm.cm_sectype_params, cm.cm_subtype_params, cm.cm_tenor_params, cm.cm_system_params, plus the SSB chain (iss_sb_subscription, iss_sb_holdings, iss_sb_redemption, iss_sb_applicant), step-up rates (iss_announcement_stepup_rates, iss_issuance_stepup_rates), participating dealers, auction safeguard, syndication allotment, etc.
- Asked: 2026-05-05
- Affects: total module count and migration-summary sequencing

### B2 — `?` U — Module ID naming convention for new modules
v0.01 uses 4-letter prefixes for the business domain (`SECMST`, `ISSCAL`, `ISSANN`, `BIDCOL`). For new modules in v0.02, suggest:
- `BANKMS` (Bank Master), `APPLNT` (Applicant), `CTRPTY` (Counterparty), `MSTCOD` (Master Code), `PUBHOL` (Public Holiday), `SECPRM` (Sectype Params), `SUBPRM` (Subtype Params), `TNRPRM` (Tenor Params), `SYSPRM` (System Params), `SSBSUB` (SSB Subscription), `SSBHOL` (SSB Holdings), `SSBRED` (SSB Redemption), `SSBAPL` (SSB Applicant), `STPRTE` (Step-up Rates), `PARTDL` (Participating Dealers), `AUCSFG` (Auction Safeguard), `SYNALT` (Syndication Allotment).
- Asked: 2026-05-05
- Affects: every new sheet's tab name and module-ID column

### B3 — `?` U — Module sequencing: parent-before-child rule
v0.01 enforces "Must run after MAS-OMEGA-X" via the Dependency column. New modules introduce new parents:
- `cm.cm_bank_master` is parent for `iss.iss_bid_*.cm_bank_master_uuid` → BANKMS-001 must run before BIDCOL-*.
- `cm.cm_master_code` and `cm.cm_master_code_category` are parents to virtually every `_code` lookup → MSTCOD must run early (block 1).
- `cm.cm_counterparty` is parent for `iss.iss_participating_dealers.cm_counterparty_uuid`.
- `iss.iss_security_master` is parent for `iss.iss_issuance.iss_security_master_uuid` (after sec→iss fold).
- Asked: 2026-05-05
- Affects: Migration Summary Seq column

### B4 — `?` U — Filter rule for cm.cm_bank_master from MEPS+ banks.txt vs ABA0009/ABA0020
Dict says `cm.cm_bank_master`'s source is "MEPS+ banks.txt". But the legacy source has `ABA0009_BANK_MASTER` and `ABA0020_STAGING_BANK_MASTER` as well. **Question:** for migration, do we read the legacy ABA0009 (one-time snapshot) or do we wait for the first MEPS+ banks.txt feed in OMEGA (interface-driven)?
